Part 1 · ~40 min

CLAUDE.md — The Standing Briefing

Make Claude read your project context before every single session.

What it is

CLAUDE.md is a file Claude reads automatically every time you open Claude Code in a project. Without it, every session starts from zero. With it, Claude already knows what you're working on, what the rules are, and what it should never do.

Think of it as a permanent sticky note on Claude's desk. You write it once and it applies to every conversation forever — until you change it.

Where it lives

LocationScopeUse it for
~/.claude/CLAUDE.mdEvery project on your machinePersonal preferences — your name, how you like responses
kp-academy/CLAUDE.mdThis repo onlyCourse context — what you're building for this program
subfolder/CLAUDE.mdThat folder onlyAdvanced — skip for now

Today you're writing the repo-level one inside your own participant folder.

Find your GitHub username first

Every file path in this session uses your GitHub username. Run this to find yours:

gh api user -q .login

It prints your username. Use that everywhere you see YOUR-USERNAME below.

Write it

Open Claude Code in your kp-academy folder:

cd ~/kp-academy
claude

Paste this prompt — replace YOUR-USERNAME with your actual GitHub username:

Create a CLAUDE.md file at participants/YOUR-USERNAME/CLAUDE.md. Read my about.md and operator/approach-memo.md first, then write four sections: (1) Who I am and what I'm working on, (2) The business problem I'm investigating, (3) The approach I've chosen and why, (4) A Don'ts section — at least 3 specific things you should never do when helping me. Each Don't should describe something you would plausibly do wrong, not something obvious like "don't break things."
Read every section before accepting. The Don'ts section especially — "never do anything dangerous" is useless. Each Don't should be specific to your project. For example: never suggest approaches that require data the business owner hasn't authorized access to, never write files outside my participant folder, never skip asking for clarification when the task is ambiguous.

Test it

Close Claude Code completely. Reopen it in the same folder. Then ask — without explaining anything first:

What are we working on and what are your guardrails?
If it answers correctly from the CLAUDE.md — commit it. If it's vague, the briefing isn't specific enough. Sharpen the first two sections until Claude can answer that question cold.
git add participants/YOUR-USERNAME/CLAUDE.md
git commit -m "Add CLAUDE.md — project briefing and guardrails"
git push
Part 2 · ~25 min

Skills — Encoding Reusable Workflows

Stop explaining the same task every session. Write it once and Claude runs it automatically.

What a Skill is

A Skill is a markdown file in .claude/skills/ inside your project. It has two parts: a description at the top (how Claude decides when to use it) and a body (what to do step by step).

The description is the most important line. It's how Claude decides whether to fire this Skill. If it's vague, the Skill never triggers. If it's specific, Claude picks it up reliably.

Example of a good description:

Use when the user asks to review the approach memo, check whether the problem statement is specific enough, or evaluate whether an approach is defensible.

Bad description: "Use for my project stuff." — too vague, will never trigger reliably.

Create your first Skill

First make the skills folder — replace YOUR-USERNAME with your GitHub username:

mkdir -p participants/YOUR-USERNAME/.claude/skills

Then paste this prompt into Claude Code:

Write a Skill called approach-review and save it to participants/YOUR-USERNAME/.claude/skills/approach-review.md. It should: read my operator/approach-memo.md, check whether the chosen path is clearly stated, whether the rejected paths have real reasons, and whether the first risk is honest — then give me specific feedback on what to sharpen. The description should trigger when I ask to review my approach, check if my memo is defensible, or get feedback on my problem statement.

Test it

Open a fresh Claude Code session. Don't type the Skill name. Just say something that matches the description naturally:

review my approach and tell me what needs to be sharper
If Claude picks it up and runs the Skill — commit it. If it doesn't fire, the description needs to be more specific. Edit it and try again.
git add participants/YOUR-USERNAME/.claude/
git commit -m "Add approach-review skill"
git push
Part 3 · ~15 min

Slash Commands — On-Demand Workflows

Same format as Skills — but you trigger them manually by typing /name.

Skills are automatic — Claude infers when to use them. Slash commands are manual — you type /name and it runs. Same file format, same folder. That's the only difference.

Use slash commands for workflows you want to run deliberately, on demand. A good first slash command: one that generates a project status update you could send to a business owner.

Create /operator-status

Write a slash command called operator-status and save it to participants/YOUR-USERNAME/.claude/skills/operator-status.md. It should read my operator/operator-log.md and operator/approach-memo.md, then generate a one-paragraph status update I could send to a business owner. Include: what I've defined so far, what approach I'm taking, and what my next question is for them. Keep it in plain English — no jargon.

Test it

/operator-status
If the output is too generic, add a line to the Skill file about your specific audience — who you'd actually send this to and what they care about. Then test again.
git add participants/YOUR-USERNAME/.claude/
git commit -m "Add operator-status slash command"
git push
Part 4 · ~10 min

Quick Hits — Handy Things to Know

ThingWhat it actually is
/clear vs new session /clear drops the conversation but keeps CLAUDE.md and Skills loaded — faster, still a clean slate. A new session (exit then claude) re-reads everything from scratch. Use /clear when context feels cluttered. Use a new session when something feels fundamentally wrong.
Shift+Tab to switch modes Cycles Claude between auto-accepting edits and Plan Mode, where it proposes a plan and waits for your okay before touching any files. Use Plan Mode for anything risky or unfamiliar; switch back to move fast once you trust what it's doing.
Esc to interrupt Stops Claude mid-task. If it's heading down the wrong path, don't wait for it to finish — hit Esc and redirect right away.
Headless mode claude -p "your prompt" runs Claude Code without opening the interactive session. You can call it from a script or chain it into a pipeline. You won't need this today — but you will in Session 6 when we connect Claude to real systems.
