The Three Tiers of Reach

How to connect Claude to anything outside files and shell

Claude Code on its own reads files and runs shell commands. To touch anything outside — your email, CRM, accounting software — you need a connector. There are three tiers. Always walk them in order.

Tier What it is When to use Effort
Tier 1
Built-in / Pre-built MCP 		Connectors already published — Gmail, GitHub, Supabase, Slack, Google Drive, hundreds more. First choice, always. 5 min install.
Tier 2
Custom MCP 		You (or Claude Code) build a small server wrapping an API your business uses. The tool has an API but no public MCP exists. 1–4 hours (Claude writes it).
Tier 3
Computer Use 		Claude controls your mouse and keyboard on screen. Tool has no API at all — legacy software, locked SaaS. Always available, but slower and costlier.
The rule: walk the tiers in order. Don't build a custom MCP if a built-in one exists. Don't use Computer Use if an API + MCP exists.

MCP: What It Is and Why It Won

The universal plug for AI agents

MCP (Model Context Protocol) is the open standard for connecting AI agents to tools and data. Anthropic created it in 2024. By mid-2025 it was adopted by OpenAI, Microsoft Copilot, Google, Cursor, Windsurf, and nearly every serious AI developer tool.

Think of MCP as USB for AI — a universal plug. Write an MCP server once; every AI platform that speaks MCP can use it. You're not building for Claude. You're building for the entire AI ecosystem.

Why this matters: learn it here, use it everywhere. Your connectors outlive any AI platform.

The MCP Ecosystem

Communication
Gmail, Slack, Outlook, Discord, Twilio
Code + Dev
GitHub, GitLab, Linear, Jira, Sentry
Data + Databases
Supabase, PostgreSQL, Airtable, Notion
Files + Docs
Google Drive, Dropbox, Confluence
CRM + Sales
HubSpot, Salesforce, Pipedrive
Finance + Ops
Stripe, QuickBooks, Zapier, n8n

Multi-Connector Demo

One prompt, two connectors, one useful output:

Find my 3 most recent client emails, cross-reference against GitHub issues mentioning their company name, then draft a follow-up message to whoever has an unresolved issue.

The power isn't one connector — it's composing them. An agent that reads email and code issues and drafts a response crosses boundaries no single SaaS tool does.

Install Your First MCP

Each pair picks one connector based on their capstone and gets real data flowing

Each pair picks one MCP based on their capstone. Common choices:

  • GitHub — code and repos
  • Gmail — email
  • Supabase — database
  • Google Drive — documents
  • Slack — notifications
  • HubSpot — CRM
  • Notion — wiki

Example: GitHub MCP

claude mcp add github -- npx -y @modelcontextprotocol/server-github

Authenticate when prompted. Generate a token at github.com → Settings → Developer settings → Personal access tokens.

Token Hygiene Rules

  • Generate with minimum scopes the MCP actually needs. Read-only research doesn't need repo-write.
  • Never paste the token into a file tracked by git. Use an env var or the MCP config file (which should be in .gitignore).
  • Write it to a password manager immediately. You'll need to rotate in 90 days.

Restart Claude Code after installing. Test with:

List my 5 most recent repos and tell me which ones I haven't committed to in a month.
Goal: every pair leaves this block with at least one real MCP connected to real data from their capstone.

Building a Custom MCP

When no pre-built connector exists — Claude Code writes the server

Most pairs won't hand-write MCP code. Claude Code builds the server; you direct it.

The Workflow

1
Pick a SaaS tool with an API — HubSpot, QuickBooks, Airtable, your internal app.
2
Find the API docs for that tool.
3
Open Claude Code.
4
Ask Claude: "Build me an MCP server that wraps these endpoints: [list 3–5 operations you care about]. Use the TypeScript MCP SDK."
5
Claude Code writes the server.
6
Add it to Claude Code's MCP list.
7
Run a task that uses the new connector.

Minimal TypeScript MCP Example

import { Server } from "@modelcontextprotocol/sdk/server";

const server = new Server({ name: "my-crm", version: "1.0.0" });

server.setRequestHandler("tools/list", async () => ({
  tools: [{
    name: "get_customer",
    description: "Fetch a customer by ID from our CRM",
    inputSchema: { type: "object", properties: { id: { type: "string" } } }
  }]
}));

server.connect();

Tool Schema Design

The schema is your prompt to Claude — write it like one

Claude reads the schema to decide whether to call the tool and how. A bad schema makes Claude guess.

Schema Element Bad Good
Tool description "Gets data" "Fetch the most recent invoice for a given customer ID. Use only when the user asks about a specific invoice; do not use for general billing questions."
Parameter description (none) "The customer's UUID, e.g. cust_4471. NOT the customer name."
Required vs optional Everything optional Explicit required/optional semantics
Field types string for everything customer_id: string, since_date: string (ISO 8601), limit: integer

Four Rules

  1. Required fields are required for a reason. If Claude could call the tool without it, it's optional.
  2. Nullable ≠ optional. Different semantics. Pick deliberately.
  3. Descriptions earn their token cost. Cut filler. Keep specifics: trigger conditions, anti-patterns, expected output.
  4. Few-shot examples belong when the right way to call the tool is non-obvious.

Computer Use

Tier 3 — when there's no API and no MCP

Computer Use lets Claude control a mouse and keyboard — either your screen or a virtual one. It sees the screen as an image, decides where to click, types, moves the mouse.

When you need it

  • Legacy software with no API
  • SaaS tools that locked down their API
  • Internal tools IT won't expose
  • Websites with no programmatic access

When to skip it

  • Any tool with an API — use Tier 1 or Tier 2 instead
  • High-volume work — Computer Use is slower and costlier per task
Computer Use is always available as a fallback. It is never the first choice. Before reaching for it, check whether an API or pre-built MCP exists.

The Integration Decision Tree

One question at a time — stop when you have an answer

When you need Claude to touch something outside files and shell:

1
Is there a built-in or pre-built MCP? → Install it, done. (Tier 1)
2
Can a shell command do it? (curl, gh, psql) → Let Claude run it. Not an MCP but often enough.
3
Does the tool have an API but no MCP? → Tier 2 — ask Claude Code to build a custom MCP.
4
No API at all? → Tier 3 — Computer Use.
5
Does it need to run 24/7, not just when you're at your laptop? → That's Session 7 (deployment).
