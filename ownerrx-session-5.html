Session 5 · Part 2 · 25 min

What Hooks Are

The difference between a suggestion and a rule

The core concept

A hook is a shell command that the Claude Code harness runs automatically in response to an event — before a tool fires, after a tool returns, or when Claude tries to end a session. The harness runs hooks regardless of what Claude decides to do. Claude cannot override a hook.

This is different from anything you put in CLAUDE.md. A CLAUDE.md instruction is a suggestion — Claude reads it and tries to follow it, but can reason its way around it in the moment. A hook is enforced at the infrastructure level. It runs no matter what.

CLAUDE.md is a suggestion. A hook is a rule.

When to use each

You want Claude to... Use...
Generally prefer a pattern Prompt / CLAUDE.md
Never do X under any circumstance PreToolUse hook
Always log when X happens PostToolUse hook
Confirm before destructive operations PreToolUse hook with human prompt
Stop a session only after a condition is met Stop hook

Where hooks live

Hooks go in .claude/settings.json under a "hooks" key. Each entry says when to fire, which tool to watch, and what command to run.

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{ "type": "command", "command": "bash /path/to/check-dangerous-command.sh" }]
      }
    ]
  }
}

Three things to notice:

  • The hook type (PreToolUse) tells the harness when to fire.
  • The matcher ("Bash") narrows it to a specific tool — you're not watching every tool, just Bash calls.
  • The command is just a bash command. Anything you can run in your terminal, you can run as a hook.
Always use absolute paths in the command field. The harness doesn't always run from the folder you expect, so ./hooks/check.sh can silently fail while /Users/yourname/project/hooks/check.sh always works.
Session 5 · Part 3 · 40 min

The Three Hook Types

PreToolUse, PostToolUse, and Stop — when and why to use each
Hook type Fires when Primary use
PreToolUse Before the tool call executes Block dangerous commands, require confirmation
PostToolUse After the tool returns its result Log every tool call, audit outputs, trigger side effects
Stop When Claude tries to end the session Force a commit, run a final verification check
PreToolUse Block dangerous commands before they run

The script receives the command as $1. Exit 1 to block. Exit 0 to allow.

#!/bin/bash
COMMAND="$1"
if echo "$COMMAND" | grep -qE 'rm\s+-rf|DROP TABLE|DELETE FROM.*WHERE 1'; then
  echo "BLOCKED: Destructive command requires human approval before running."
  exit 1
fi
exit 0
Exit 1 blocks. Exit 0 passes. Hooks that crash and implicitly exit 0 fail open — meaning they silently allow everything through. Always test your hook script directly in the terminal before wiring it in. Run it against a safe input and a dangerous input and confirm you get the behavior you expect.
PostToolUse Log every tool call to an audit file

The script receives the tool name as $1, the input as $2, and the output as $3. Append to a .jsonl file for easy review later.

#!/bin/bash
TOOL_NAME="$1"
INPUT="$2"
OUTPUT="$3"
TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
echo "{\"timestamp\": \"$TIMESTAMP\", \"tool\": \"$TOOL_NAME\", \"output_length\": ${#OUTPUT}}" >> audit.jsonl
Once you have an audit log, you're not guessing what your agent did. You're reading the receipt. When something goes wrong in production, this file is the first thing you open.
Stop Auto-commit work before the session ends

Runs when Claude attempts to end the session. Use it to enforce hygiene — like making sure work is always committed before Claude walks away.

#!/bin/bash
cd "$PROJECT_DIR"
if [ -n "$(git status --porcelain)" ]; then
  git add participants/
  git commit -m "auto: session work $(date +%Y-%m-%d)"
fi
Stop hooks that fail (exit non-zero) can prevent Claude from ending the session cleanly — the session gets stuck in a loop. Always test your Stop hook in isolation before wiring it in, and make sure every code path exits 0.

Structured Error Responses

Stop making Claude guess what went wrong

When a tool returns a plain string like "Error: something went wrong", Claude has to guess whether to retry, stop, ask for help, or try a different approach. A structured error with a category removes the guesswork entirely — Claude reads the category and knows exactly what to do.

The four error categories

Category Example cause What Claude should do
retryable Network timeout Wait the specified delay, then retry (up to max retries)
permanent Record not found Do not retry — report the failure and move on
authorization Expired token, permission denied Stop immediately — get a human to resolve credentials
schema Required field missing Re-prompt Claude to correct the input and try again

Before: a plain string error

return "Error: something went wrong with the database"

After: a structured error

return {
  "error": True,
  "category": "retryable",
  "message": "Database connection timed out after 5 seconds",
  "hint": "Retry after 2 seconds. Maximum 3 retries before escalating.",
  "retry_after": 2
}

Permanent error example

return {
  "error": True,
  "category": "permanent",
  "message": "Patient record ID 4821 does not exist in the system.",
  "hint": "Do not retry. Report this record ID as not found."
}

Authorization error example

return {
  "error": True,
  "category": "authorization",
  "message": "API token expired. Request rejected with 401.",
  "hint": "Stop and alert a human to refresh credentials before continuing."
}
The hint field is optional but powerful. It lets you encode your team's decision logic directly into the error — so Claude follows your protocol, not its default instinct.

Validation-Retry Loops

Catch bad outputs before they become bad actions

A validation-retry loop sits between a tool call and everything downstream. It checks the output against a schema before letting Claude act on it. If the output is invalid, it returns a schema error and Claude retries with corrected input. If it's still invalid after two retries, it escalates instead of continuing.

The pattern

1
Tool call — Claude calls the tool with its current input.
2
Validate output — Check the result against your expected schema. Is every required field present? Are values the right type and range?
3
If invalid — Return a structured error with "category": "schema" and a specific hint about what was wrong. Claude reads the hint and retries.
4
Two retries max — If the output is still invalid after two attempts, stop and escalate. Never loop forever.
Always put a hard cap on retries. A loop with no exit condition will keep Claude running indefinitely and rack up cost. Two retries then escalate is a reasonable default for most tools.

Exercise

Each pair wires a validation-retry loop around the riskiest tool in their pipeline — the one where bad output would cause the most downstream harm. Use the structured error format from Part 4 for the schema error response. Commit to the capstone lead's folder when done.

Your schema is based on mock data. The field names, types, and value ranges you validate against today were picked when your data was made up. When real sponsor data arrives next week, this validation loop is the first thing to revisit — real inputs may have different structures, unexpected nulls, or formats you didn't anticipate. The loop won't break, but it may be validating the wrong things. Treat this as version 1.
Session 5 · Part 6 · 10 min

The Hookify Pattern

When you've corrected Claude three times, write a hook instead

Here's a simple heuristic: if you've given Claude the same correction three times in prompts — "don't do X," "remember not to do X," "I said don't do X" — stop correcting and write a hook instead.

The third time you say something in a prompt, that's a PreToolUse hook waiting to be written. Prompts are for preferences. Hooks are for rules. Once you recognize that something must never happen, it belongs in a hook — not in a prompt you'll have to repeat next session.

The pattern: Notice the repetition → write the hook script → add it to settings.json → delete the correction from your prompt. Your agent gets more reliable, and your prompts get shorter.

As you build out your capstone over the coming sessions, keep a short list of things you keep having to remind Claude about. That list is your hook backlog.
