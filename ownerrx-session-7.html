Part 1 · 10 min

MCP Quick Review

Where we left off — before we talk about where your agent lives

Last session covered how your agent connects to the outside world. Quick recap before we move to deployment.

MCP = "USB for AI"

MCP (Model Context Protocol) is a universal plug standard. Instead of building a custom connection for every tool, you plug in once and any MCP-compatible service just works — the same way USB lets you plug any keyboard into any computer.

Three Tiers of Reach

Tier What it is Time to set up
Tier 1 Pre-built MCP — someone already built the plug (Gmail, Google Drive, Notion, HubSpot) 5 minutes
Tier 2 Custom MCP via API — your sponsor has an API, you build the plug 1–4 hours
Tier 3 Computer Use — Claude watches and controls the screen directly Last resort only

5 Questions to Ask Your Sponsor

Before you can connect your agent to real data, you need answers to these:

  1. Where does the data live? (spreadsheet, CRM, database, email)
  2. Is there an API? (if yes → Tier 2)
  3. What format is the data in? (what columns, fields, how it's structured)
  4. How often does it update? (real-time, daily, weekly)
  5. Who controls access? (do they need to grant permission before you can connect)

Part 2 · 10 min

Why Deploy?

The difference between a demo and a product

Your agent lives in your terminal. That works for interactive tasks. But real agents need to:

  • Run on a schedule — every Monday at 7am, generate the weekly brief
  • React to events — intake form submitted → agent drafts proposal
  • Serve a web interface — team asks questions from Slack or browser
  • Scale beyond one laptop

That means putting it somewhere always-on.

A tool that only runs when you're watching it is a demo.
A tool that runs on its own is a product.
Deployment is the line between those two things.

The Modern Trio

Three platforms handle ~80% of what small-business agents need
Vercel 8 min

What it is

The platform for web front-ends and serverless APIs. Push to GitHub, Vercel deploys automatically.

When to pick it

  • Your agent has a web interface or API
  • You want preview URLs on every pull request
  • Short-running tasks (under 5 minutes)
  • Zero-config deploys matter to you

Cost

Generous free tier. Paid plan is $20/month.

One-line deploy

npx vercel
Railway 8 min

What it is

Platform for long-running services, workers, and persistent processes.

When to pick it

  • Your agent polls for data or listens on a queue
  • Needs a process longer than 5 minutes
  • You want a database sitting right next to your code

Cost

~$5–20/month for small projects.

Deploy steps

1
Go to railway.app and sign up with GitHub.
2
Click New Project → Deploy from GitHub repo.
3
Pick your repo. Railway auto-detects your stack and deploys.
Supabase 9 min

What it is

Postgres database + auth + storage + edge functions, bundled. Open source.

When to pick it

  • You need a database (almost always)
  • You want auth without building it yourself
  • You need file storage

Cost

Free tier is meaningful. Pro plan is $25/month.

The combo that runs most modern AI products:
Vercel or Railway for your agent's code + Supabase for its memory.

Part 5 · 25 min

Security Essentials

Four concrete areas — do this / don't do that

You're building agents that will touch email, CRM, payments, and databases. This section is practical.

1. Token Exposure (8 min)

  • Never commit a token to git. Even private repos — GitHub scans for them.
  • Never paste a token into Claude Code when you could use an env var. Logs get compacted and shared.
  • Always add .env* to .gitignore.
  • Rotate tokens quarterly. When someone leaves your team, rotate immediately.
  • Scope tokens to the minimum permissions needed.

Live check — run this in your repo

git log --all --full-history -p | grep -iE "sk-ant-|ghp_|sk_live_|AKIA" | head
If anything comes up: rotate those keys immediately, then look up git-filter-repo to scrub history.

2. AI-Generated Code Review (6 min)

What to watch for in code Claude writes:

  • Silent failures: try/except: pass swallows errors. Claude sometimes adds these "defensively." They hide real bugs.
  • SQL built by string concatenation: classic injection. Use parameterized queries.
  • Over-broad permissions: Claude scaffolds Supabase policies that default to "everyone can read everything." Tighten before shipping.
  • Shell commands built from unchecked user input: remote command execution bug. Use subprocess with a list, not a formatted shell string.
  • Dependencies you've never heard of: Claude sometimes invents package names. Check every import against a real registry.

3. Row Level Security — Supabase (5 min)

RLS is a policy that says even authenticated users only see rows they're allowed to see. Without RLS, any user with valid auth can read every row in your database.

Default: RLS is OFF on new tables until you turn it on. Many data leaks happen because someone forgot this.
  • Enable RLS on every table that holds business data.
  • Policies are per-table, per-operation.
  • Default to deny. Test with a non-admin user.
alter table capstone_runs enable row level security;

create policy "users read own runs"
on capstone_runs for select
using (auth.uid() = user_id);

4. Claude Code Permissions (6 min)

  • Use Plan Mode when you're not sure what Claude will do.
  • Your hooks from Session 5 earn their keep here — a PreToolUse hook can block destructive commands entirely.
  • Scope MCPs minimally. Gmail MCP for drafting doesn't need send permission. GitHub MCP for reading doesn't need repo-delete.
  • Run risky work in sandboxes — Codespaces, worktrees, test accounts.
  • Before letting Claude do something risky (delete files, push to prod, message customers), ask it to show you the exact command first.

Minimum Viable Security Posture

  1. No secrets in git — verify with the grep scan above.
  2. RLS on every table holding business data.
  3. Tokens scoped minimally, rotated quarterly.
  4. AI-generated code reviewed before merging anything that touches data, auth, or external APIs.
  5. One security-focused antagonist prompt in your pipeline.
Part 6 · 20 min

Incident Response

Every deployed agent will break — the question is whether you respond professionally

The sequence. Memorize it.

1
Triage

Is a user blocked right now? Who's affected? What exactly is broken vs. what might be broken? One sentence each. Don't debug yet.

2
Contain

Stop the bleeding. Disable the agent, roll back to last-known-good, put up a "temporarily unavailable" state. Contain before you diagnose.

3
Diagnose

Use logs, not guesses. Find the specific run that failed, the specific input, the specific stage. If you can't reproduce from logs, your logs aren't good enough.

4
Communicate

Tell your sponsor before a user tells them. One message: what happened, who's affected, what you did, when you'll have more info.

5
Fix

Smallest change that addresses the root cause. Not a rewrite. The fix should be narrow enough to review in five minutes.

6
Prevent

Add an eval case that would have caught this specific failure. If your eval set wouldn't have caught it, the incident wasn't random — it was a gap in your process.

The instinct is "It's broken → fix it."
The senior move is: contain first, communicate early, fix narrow, prevent permanently.
