The Four Primitives

Every modern AI platform is built from these four things.
PrimitiveWhat it isWhat it gives you
SkillA saved Markdown file that tells Claude how to do something when a trigger matchesRepeatability — your best prompts become reusable workflows
PluginA bundle of skills, connectors, and subagents shipped as one installable packageDistribution — share an entire workflow with your team
SubagentA fresh Claude instance spawned for a sub-task, with its own context windowParallel speed + specialization
ConnectorA pipe to external data or tools — Gmail, GitHub, Supabase, your CRMReach beyond files on your disk

Every AI product you'll use or build — Salesforce Einstein, Microsoft Copilot, your company's internal chatbot — is some combination of these four things. You now have the vocabulary to describe any AI system.
Part 3 · 30 min

Skills: Teaching Claude Your Work

Encode how you want things done — once — and every future session follows that pattern.
Concept5 min

A skill is a Markdown file in .claude/skills/. When you give Claude a prompt, it reads the description of every skill it has access to. If your prompt matches, that skill's instructions load automatically before Claude responds.

Anatomy of a good skill

---
name: weekly-brief
description: Use when the user asks for a weekly summary, Monday brief, or this-week roundup
---

When generating a weekly brief:
1. Read `notes/` for anything new this week
2. Structure output as: Highlights / Decisions / Risks / Next Week
3. Keep each section to 3 bullets max
4. End with one "ask" — something I need to decide

Quality criteria:
- Every bullet names a specific thing (no "various updates")
- The "ask" is actionable this week, not aspirational
1 — Context
What does Claude need to know? Which files, what project, who the audience is.
2 — Instructions
The steps in order. Specific enough for a new employee following a checklist.
3 — Output format
Exactly what the result should look like — headers, length, tone.
4 — Quality criteria
What makes this output good vs. bad? The self-check Claude runs before returning.
The description field is the most important line. It's not a label — it's a trigger. Describe when to fire, not what it does.

Good: "Use when the user asks for a weekly summary, Monday brief, or this-week roundup"
Bad: "This is a skill for writing weekly summaries"

Each pair creates one skill for their capstone. Work in the capstone lead's folder. This skill should encode a workflow you'll run repeatedly.

Part 4 · 15 min

Plugins: Shipping a Whole Workflow

A plugin bundles related skills, subagents, and connectors into one installable package.

A skill is one workflow. A plugin is a collection of workflows plus the agents and connections they depend on — bundled so the whole thing installs in one step.

my-sales-plugin/
├── skills/
│   ├── qualify-lead.md
│   ├── draft-followup.md
│   └── score-deal.md
├── agents/
│   └── competitive-researcher.md
├── mcp/
│   └── hubspot-connector.json
└── plugin.json

A plugin is just a folder of files — no binary, no compiled code. It lives in GitHub. Version it, update it in one place, everyone gets the update.

Skills are for you. Plugins are for your team. A plugin is the unit of distribution.
A plugin marketplace is emerging. Internship projects that get published will be the ones with real business value — something to keep in mind as you design your capstone.
Part 5 · 20 min

Subagents: Parallel Work

A subagent is a fresh Claude instance spawned for a sub-task, with its own context window.
1
Parallel speed. Research 5 competitors simultaneously instead of sequentially. A 10-minute task might take 2 minutes with 5 subagents running at once.
2
Context protection. Long tool outputs stay inside the subagent. The main thread stays focused on high-level decisions.
3
Specialization. Each subagent has its own system prompt. Your research subagent can be ruthlessly concise. Your writing subagent can match your brand voice.
Live Demo10 min
Research these three competitors in parallel using separate subagents for each: [Company A], [Company B], [Company C]. For each: what they do, who their target customer is, and one thing they do better than everyone else. Give me a comparison table when all three are done.

Watch Claude spawn three parallel tasks. Then run this counter-example:

Read README.md and tell me what it says.

Claude doesn't spawn a subagent for this. Subagents have overhead — use them when work is independent and chunky.

Common mistake — over-spawning: Don't spawn a subagent to read one file or answer one question. Ask: "Could I do these tasks in parallel if I had more hands?" If yes, subagents help. "Does each task need the previous one's result?" If yes, use a pipeline.
Part 5B · 25 min

Loop Mechanics, Hub-and-Spoke & Session Resumption

How the agent loop actually works — and why it sometimes doesn't stop when you want it to.
Concept 1 — The Agent Loop5 min
user prompt → model decides → tool call → tool result → model decides → ...                                                             ↓                     (returns final response — no more tool calls)
1
Why does the loop terminate? Because the model produces a turn without a tool call. No tool call = "I'm done." This is how every agent in every framework works.
2
Why does it sometimes NOT terminate? Because the tool keeps returning data the model interprets as needing more action. Open-ended goals ("keep improving until it's perfect") may never terminate. Always define what "done" looks like.
Common mistake: "Research this topic thoroughly" invites an infinite loop. "Research this topic and give me the top 5 findings, then stop" gives the loop a clear exit condition.
Concept 2 — Hub-and-Spoke vs Pipeline8 min
ShapePatternWhen it wins
PipelineA → B → C, each step feeds the nextSequential workflows. Can't route until you classify. Can't classify until you extract.
Hub-and-spokeOne orchestrator delegates to N specialists, then collectsParallel work; or when the next step depends on a combination of all outputs.
Most real-world workflows combine both shapes. Learn to see them and you can describe any agent architecture to anyone.
Concept 3 — Session Resumption5 min
claude --resume <conversation-id>
Use resumption when
You walked away from a long-running task. Pick up exactly where you left off, full context intact.
Don't use resumption when
Starting a new unrelated task. Stale context bleeds in and produces worse results. Use /clear instead.
Concept 4 — Provenance5 min

When work flows through multiple agents, you need to trace any output back to where it came from. Every agent output should carry a small header:

# Agent: research-agent
# Prompt version: v1.2
# Input ID: ticket-2026-0042
# Timestamp: 2026-06-04T14:32:00Z
Provenance is not optional for production agents. Without it, debugging a 5-agent pipeline is guessing. With it, you know exactly which agent produced the bad result and which prompt version was running.
Part 6 · 25 min

Pipelines + Antagonist Evaluators

Specialized agents handing off to each other — with an adversarial agent that catches what the generator missed.

A single agent has blind spots — it generates output and evaluates its own output with the same biases. Pipelines fix this: specialized agents hand off, with an antagonist whose entire job is to find problems.

Research Agent → Draft Agent → Antagonist Evaluator gathers context produces output critiques, scores, flags weaknesses                                                     ↓                             (if score below threshold → loop back to Draft)
  • The generator agent tries to produce output that satisfies the prompt. Its bias is toward completion and confidence.
  • The antagonist agent has a different system prompt: "Find everything wrong with this. Score it. Flag every weakness." Its bias is toward criticism.
  • Two opposing biases produce better results than one agent evaluating its own work.
This is why code review exists. And peer review. And red teams. The pattern is ancient — you're encoding it in software. The generator wants to be right. The antagonist wants to find the error. You need both.
Watch for this: A weak antagonist will just rephrase the draft instead of critiquing it. Your homework evaluator checks for exactly this. An antagonist that agrees with everything is useless.
Live Build15 min

Each pair builds a 3-agent pipeline for their capstone task. Work in the capstone lead's folder.

I want to build a 3-agent pipeline for my capstone task: [describe your task].

Create three agent files in `.claude/agents/`:
1. `research.md` — focused on gathering context and raw information
2. `draft.md` — focused on producing structured output from the research
3. `antagonist.md` — focused on finding problems, not rephrasing the draft

Each file should have: a name, a system prompt for its specific role, and instructions on what it receives and what it should produce.

Then write a skill in `.claude/skills/run-pipeline.md` that orchestrates all three in sequence.
1
Read the three agent files. Is each system prompt focused on one job? A research agent that also tries to write should be split.
2
Run the pipeline on a real piece of data from your capstone domain. Watch the handoffs.
3
Run the same task without the pipeline. Compare outputs. What did the antagonist catch?
4
Check your antagonist. Ask Claude: "Did it find a real problem, or just rephrase the draft? What's the evidence?"
Part 7 · 5 min

Quick Hooks Preview

Session 5 is entirely about hooks. Here's what's coming.

Hooks are scripts Claude Code runs automatically at specific events — not because you asked, but because you configured it to. They turn "Claude should do X" from a suggestion into a rule.

The difference between skills and hooks: A skill tells Claude how to do something when asked. A hook fires automatically at a specific event, whether you asked or not. Skills are about quality. Hooks are about enforcement.

If you're curious before Session 5, ask Claude Code: "What hooks does Claude Code support and how do I configure them?"
