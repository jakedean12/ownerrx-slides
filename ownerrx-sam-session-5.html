Step 0 — Prerequisite
Build a Quick Eval Set · ~15 min
Catch-up from last time

You need a regression baseline before you harden anything

Every hook you add tonight risks breaking the happy path. An eval set is how you prove it didn't.

Create participants/sammpentz-commits/artifacts/evals/cases.md with 5 realistic test cases. Each one is input data plus what the pipeline should do with it.

1
Clean baseline — a normal student with assignments, grades, and study history. Should pass straight through.
2
No study history — a subject with zero past study logs. The draft agent should recommend a general-purpose method and say so — not invent a method that "worked before."
3
No free time — a packed calendar with no open windows. The plan should say so, not invent free time.
4
Overdue assignment — something already past its due date. Should rank at or near the top.
5
Adversarial — deliberately missing a grade field. The pipeline should say "not provided," never guess a grade.
5 is the minimum for tonight, not the finish line. Grow this to 10–20 before Session 6 — that's the official course bar, and it's the regression set you'll run against every future change.
Exercise 1
PreToolUse Hook · ~20 min
The wall Claude can't argue past

Block a study block that overlaps a real event

A prompt is advice Claude can override. A hook is a wall it can't see past.

Your CLAUDE.md already says: never auto-move or delete a calendar event, and the draft agent should only use free time. Right now that's enforced by Claude reading the rule and agreeing with it. Tonight it becomes a hook.

The rule, in one sentence: A proposed study block must not overlap a "busy" slot from the research agent's calendar data, and must not fall between 11pm–7am.

Step 1 — Ask Claude to write the hook

I want a PreToolUse hook that protects my Planana study planner pipeline. Before any study block gets written into the final plan, check it against the research agent's free-time windows.

Write a bash + jq script saved to participants/sammpentz-commits/.claude/hooks/check-study-block.sh that:
1. Reads the tool call payload from stdin
2. Pulls the proposed study block's start/end time
3. Blocks (exit 2) if the block overlaps any "busy" window from the research data, or falls between 11pm and 7am
4. On block, writes a specific stderr message naming the exact conflict — not just "blocked"
5. Exits 0 if the block is clean

Then wire it into .claude/settings.json with the right matcher.

Step 2 — Read the script before testing it

Walk through it line by line. Find: where does it read the payload? Where does it decide busy vs free? What's the exact stderr message on a block?

Step 3 — Test both directions

1
Feed it a study block that deliberately overlaps an existing event — confirm it's blocked with a useful message.
2
Feed it a block in a genuinely free window — confirm it passes.
3
Feed it a block at 1am — confirm it's blocked for the sleep-hours rule, not just the overlap rule.
If it blocks everything, the matcher's too broad. Narrow it to the specific tool call that writes the final study block — not every tool call in the pipeline.

Exercise 2
PostToolUse Hook · ~15 min
Observe, never block

Log every study plan delivered

PreToolUse is a wall. PostToolUse is a witness — it records what happened after the fact.

Lower stakes than a hospital or fintech example, but the habit matters: once real students use this, you'll want a record of what was recommended and whether the antagonist had to step in.

Build it

Write a PostToolUse hook saved to participants/sammpentz-commits/.claude/hooks/log-study-plan.sh that fires after the pipeline produces a final study plan. It should append one JSON line to participants/sammpentz-commits/audit/audit.jsonl with: timestamp, which assignments were ranked, which study methods were recommended, and whether the antagonist passed or failed (and how many revision rounds it took).

Make sure it always exits 0 — this hook observes, it never blocks. Add audit/ to .gitignore so the log contents never get committed, only the hook script and settings.json config.

Test it

Run the pipeline once on real sample data. Open audit.jsonl and confirm one line appeared with every field populated.

Why this matters later: once Planana has real students, this log is how you answer "how good are our recommendations, actually?" without needing to ask anyone.
Exercise 3
Structured Error Categories · ~15 min
Give Claude something to act on

Turn "say so, don't guess" into a real category

A tool that returns "something went wrong" makes Claude guess. A category + a hint makes it recover correctly.

Your research agent already has a rule: if a field is missing, write "not provided," never guess. Today that becomes two real categories instead of one prose sentence.

CategoryExample for your pipelineWhat the agent should do
SchemaA calendar event is missing an end timeFlag it, don't process that event as a free/busy block
PermanentAn assignment's due date can't be parsedDon't retry — surface it, ask for the real date
Skipping retryable and authorization for now — your pipeline doesn't call any live external API yet, so there's nothing transient to retry against. That changes once Session 6 wires in real Google Calendar / Planana database sync.

Build it

Update the research agent (participants/sammpentz-commits/artifacts/pipeline/research.md) so that when it encounters bad input data, it returns a structured error instead of a plain sentence:

{"category": "schema", "message": "Calendar event 'Math class' is missing an end time", "hint": "Cannot determine if this slot is free or busy — ask the student to confirm the event's end time."}

{"category": "permanent", "message": "Assignment 'Lab Report' has an unparseable due date: 'next wknd'", "hint": "Ask the student to provide the due date in a standard format."}

Use "schema" for malformed/missing calendar or assignment fields, and "permanent" for anything that's invalid and won't fix itself on retry.

Test it

Force each category: feed it a calendar event missing an end time, then an assignment with an unparseable date. Confirm the category and hint are both useful — not just "error."

Exercise 4 — The Big One
Validation-Retry Loop · ~15 min
From prompt to enforced rule

Hard-code the "max 2 revision rounds" rule

This is the payoff. The thing you've already designed becomes the thing the harness guarantees.
draft agent → antagonist check → PASS? → deliver ↓ FAIL feed specific failures back to draft ↓ re-run antagonist (round 2 of 2 max) ↓ still FAIL → deliver best draft + note unresolved issues

Right now this lives as five sentences in run-pipeline.md. Today it becomes a real cap that can't silently be skipped or extended.

Build it

Update run-pipeline.md so the draft → antagonist resolve step is an explicit, hard-capped loop, not a prose instruction:

1. Run draft, then antagonist.
2. If PASS → deliver, done.
3. If FAIL → feed the antagonist's specific failures back to draft, re-run draft (revision 1), re-run antagonist.
4. If still FAIL after revision 1 → one more round (revision 2) — hard cap, no exceptions.
5. If still FAIL after revision 2 → deliver the best draft so far, with a visible note: "Pipeline: 2 revisions attempted, X issues remain unresolved: [list]."

Write this as numbered, unambiguous steps with the cap stated as a number (2), not a word like "a few." Also log the revision count to audit.jsonl from Exercise 2.

Test it

Run your 5 eval cases from Step 0 again. Confirm:

1
The clean baseline case still passes on the first round — no regression.
2
At least one case (try the adversarial one) triggers a FAIL → revision → re-check, and you can see it happen.
3
If you can force two FAILs in a row, confirm it stops at round 2 and delivers with a clear unresolved-issues note — it does not loop a third time.
This is the exercise that matters most tonight. A pipeline that retries forever, or skips its own evaluator under pressure, is the kind of thing that quietly burns a company's API budget or ships a bad plan to a real student. Today yours can't do either.
