Build /operator-status

A slash command Sam triggers manually to generate a status update on the Planana capstone.

A slash command is identical to a skill file — same folder, same format. The only difference: a skill fires automatically when Claude detects the right trigger. A slash command fires when Sam manually types /name.

Analogy: A skill is a smoke detector — fires on its own when it detects something. A slash command is a light switch — you flip it when you want it.

Loop Mechanics · ~15 min
Concept — no building

How Claude Code actually works under the hood

Understanding the loop explains why pipelines work and why agents sometimes go off the rails.

The loop

User prompt → Claude decides → Tool call → Tool result → Claude decides → ...                                                           ↓                              (no more tool calls = done)

The loop ends when Claude produces a response without calling any tools. That's the signal it thinks it's done. Everything in between — reading files, writing code, running commands — is just Claude going around this loop until it decides the job is finished.

Why this matters for Planana

Sam's pipeline is going to: read assignment data → generate a study plan → evaluate it. That's three tool calls in a loop. Without a clear stopping point, Claude might keep "improving" the plan forever.

Bad prompt — infinite loop: "Keep improving the study plan until it's as good as possible." Claude has no idea what "as good as possible" means. It will loop indefinitely.
Good prompt — clear exit: "Generate one study plan, evaluate it once for conflicts and hallucinated data, return the final version, then stop." Claude knows exactly when it's done.

The rule: always define what "done" looks like. If you can't describe the finished output, Claude can't decide when to stop either.

Pipeline vs Hub-and-Spoke

ShapePatternWhen to use it
PipelineA → B → C, each step feeds the nextEach step depends on the previous one. Can't draft before researching. Can't evaluate before drafting.
Hub-and-SpokeOne orchestrator sends to N specialists at once, collects resultsSub-tasks are independent — can run in parallel. One agent reading math assignments while another reads history assignments.
For Planana tonight — Pipeline wins. Sam can't generate a study plan before reading assignments. He can't evaluate the plan before it exists. The data flows one direction: gather → generate → evaluate. Sequential all the way through.

Session Resumption

If Sam is mid-session and has to stop, he can pick up exactly where he left off:

claude --resume <conversation-id>

Claude reloads everything — every file it read, every tool it called. Use it for genuinely paused work. For a new task, always start fresh with /clear — old context bleeds in and confuses Claude.

Session 4 — Main Build
3-Agent Pipeline · ~35 min
Build time

Research → Draft → Antagonist

Three specialized agents handing off to each other, with an adversarial agent that catches what the generator missed.

The concept

A single agent has blind spots — it generates output and evaluates its own output with the same biases. A pipeline fixes this: each agent has one job and no investment in the other agents' work.

Research Agent → Draft Agent → Antagonist Evaluator reads assignments generates study plan finds real problems grades, study logs method recommendations flags hallucinated data calendar data suggested time blocks flags calendar conflicts

The antagonist works because it's given a completely different instruction — not "generate a good plan" but "find everything wrong with this plan." It has no reason to be generous.

Analogy Sam will get immediately: Submitting a draft assignment vs having a friend tear it apart first. The friend who finds your mistakes before your teacher does is the antagonist agent.

Step 1 — Build the pipeline

Paste this into Claude Code:

I want to build a 3-agent pipeline for my Planana capstone. The agent analyzes a student's assignments and calendar to generate a personalized study plan.

Create three agent files in participants/sammpentz-commits/artifacts/pipeline/:
1. research.md — gathers the student's assignments, grades, and past study logs
2. draft.md — generates a ranked study plan with method recommendations and suggested time blocks
3. antagonist.md — checks that every recommendation references real student data, no hallucinated assignments or grades, and all suggested blocks fit in actual free calendar time

Then write a skill in participants/sammpentz-commits/.claude/skills/run-pipeline.md that orchestrates all three in sequence: research → draft → antagonist → final output.
Don't interrupt while Claude builds. Let it finish all four files before touching anything. It may take 2–3 minutes.

Step 2 — Read the agent files

Before running anything, read all three files Claude created. For each one ask: "Is this agent focused on exactly one job, or is it trying to do too much?"

A research agent that also tries to draft is wrong. Split it. A draft agent that also evaluates is wrong. Split it. Each file should be doing exactly one thing.

Step 3 — Run it on Planana sample data

Run the pipeline using this sample student data:

Assignments:
- Math test on Friday (subject: math, grade so far: B-)
- History essay due Thursday (subject: history, grade so far: A-)
- Science homework due tomorrow (subject: science, grade so far: B+)

Past study logs:
- Math: flashcards, 2 hours, got B on last test
- History: reading notes, 1.5 hours, got A- on last essay

Calendar free time:
- Tonight: 7pm - 9pm
- Tomorrow: 4pm - 6pm
- Wednesday: 5pm - 8pm

Step 4 — Compare single agent vs pipeline

Run the same request without the pipeline — just a direct prompt:

Given these assignments and calendar, generate a personalized study plan with method recommendations and suggested time blocks: [paste the same student data]

Compare the two outputs. What did the antagonist catch that the single-agent version missed?

Step 5 — Check the antagonist

Did the antagonist agent actually find a real problem in the draft, or did it just rephrase what the draft said? What's the evidence?
If the antagonist was too nice — it just agreed with the draft — add this to its system prompt: "Be adversarial. Find every flaw. If a recommendation references data that isn't in the student's profile, flag it explicitly. If a study block conflicts with a calendar event, flag it. If the plan looks good, you're not looking hard enough." Then run the pipeline again.
